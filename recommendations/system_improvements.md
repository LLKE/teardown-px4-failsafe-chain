## System Improvements

To avoid chosing a failsafe action based purely on implicit control flow, there should be an explicit policy table, such as a map that maps system failures to an enum of actions, that encodes which system failure is allowed to trigger which failsafe action. The enum values should not rely on ordering, but on explicit numbering. This way, adding a new failsafe action needs explicit decisions by the developer, for which system failure to allow this new action. 

The decision process should not be split over multiple locations, such as `from...Act` directly after the failsafe check, to the fallthrough chain selecting higher order severity actions in case the chosen action cannot run. This should be unified into one location, e.,g. the `from...Act` function. This function could explicitly iterate through the suitable actions using a for-loop, instead of a switch-case fallthrough chain to ensure no regressions can occur if the chain is modified (and to reduce boilerplate).
